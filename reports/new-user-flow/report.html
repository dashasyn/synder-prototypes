🚨 Top 3 Critical Issues

Critical   #1 — Role selection drop-off (63.9%)

The "Choose an option" step on /onboarding/index loses 64% of all users. Users see a role/product selection screen with unclear options and no context about what choosing differently means for their experience.

Recommendation: Simplify to a single question with clear outcome descriptions. Consider auto-detecting user type from signup data (e.g., Stripe = SaaS, Shopify = ecommerce) or deferring this choice entirely.

Critical   #2 — TypeError blocking Mapping page

"TypeError: Cannot read properties of undefined (reading 'email')" — affects 3+ user sessions, still active as of today. Completely blocks new users from proceeding past the currency/mapping configuration step.

Recommendation: Immediate hotfix needed. Add null-check for email property in mapping component. This is a P0 bug blocking onboarding completion.

Critical   #3 — No clear "what happens next" after signup

The journey from marketing site → signup → onboarding has no progress indication, no welcome context, and immediately drops users into a role selection screen without explaining the 3-step process (Connect → Import → Reconcile).

Recommendation: Add a welcome screen after signup showing the 3-step journey, estimated time, and what they'll need (accounting software credentials, payment platform access).

4. Friction Log

Critical   F-01: Role selection abandonment

Screen: /onboarding/index · Element: "Choose an option" dropdown

63.9% of users never interact with this element. The label "Choose an option" provides zero context. Users don't know what they're choosing or what the consequences are.

Rec: Replace with a visual card-based selector showing 2-3 clear paths with icons, descriptions, and "best for..." labels. Add a "Not sure? Start here →" default option. Consider removing this step entirely and auto-detecting from signup context.

Critical   F-02: JS error blocks Mapping page

Screen: Mapping/currency configuration · Error: TypeError: Cannot read properties of undefined (reading 'email')

Active bug affecting 3+ user sessions, still reproducing as of March 20, 2026. Completely prevents new users from completing onboarding. Likely caused by missing user object data in mapping component.

Rec: P0 hotfix. Add defensive null-checks. Investigate why user.email is undefined at this point — possible race condition or incomplete signup data propagation.

Critical   F-03: No progress indication anywhere in onboarding

Screen: All onboarding steps

There is no step counter, progress bar, or "Step X of Y" indicator. Users have no idea how long onboarding will take or where they are in the process. The marketing site promises "easy to set up" and "three simple steps" (Connect → Import → Reconcile) but the in-app experience doesn't reflect this.

Rec: Add a persistent progress stepper showing: 1) Choose setup → 2) Connect platforms → 3) Configure mapping → 4) Import data → 5) Start syncing. Show estimated time ("~5 minutes").

Major   F-04: No welcome/context screen after signup

Screen: Transition from signup → onboarding

Users go directly from creating an account to a role selection screen. No "Welcome to Synder" moment, no explanation of what to expect, no setup checklist. The marketing site's "Build your month-end bookkeeping in three simple steps" promise is invisible in the actual product.

Rec: Add a welcome interstitial: "Welcome! Here's what we'll set up together" with the 3-step visual from the marketing site, estimated time, and a "Let's go" CTA.

Major   F-05: Fragmented onboarding paths

Screen: /onboarding/index vs /revenueRecognitionOnboarding/index

Synder Sync and RevRec have completely separate onboarding flows at different URLs. Users who pick the "wrong" product at step 1 have no easy way to switch. There's also no indication during onboarding that other products exist or that they can add them later.

Rec: Unify into a single onboarding flow that branches based on product selection but shares the same UI shell, progress bar, and navigation. Add "You can add more products later" reassurance.

Major   F-06: Product Mappings JSON serialization error

Screen: Product Mappings page · Error: TypeError: Converting circular structure to JSON

Occurs when users interact with Product Mappings. The app attempts to serialize a DOM element, indicating a bug where a DOM node reference is stored in state/data that gets JSON-stringified.

Rec: Fix the serialization bug. Ensure only plain data objects (not DOM nodes) are stored in state that gets stringified. Add error boundary to prevent white screen.

Major   F-07: Pricing inconsistency between marketing and product

Screen: synder.com/pricing/ vs synder.com/industry/syndersync/

Plan names differ across pages: "Starter/Medium/Scale/Enterprise" on main pricing vs "Essential/Pro" on product pages. Price anchors also differ ($115/mo vs unclear tier pricing). This erodes trust during the critical pre-signup decision moment.

Rec: Audit all pricing references across the site for consistency. Standardize on one set of plan names. Add a pricing calculator that shows exact price based on transaction volume.

Major   F-08: No guided post-onboarding experience

Screen: Post-onboarding dashboard/landing

LogRocket data shows first-session users scattering to Settings, Transactions, Control panel, Invoices, Insights, Dimensions, and Rules with no clear pattern — suggesting they don't know what to do after onboarding completes. No "success" moment or guided next step.

Rec: After onboarding completion, show a success screen with a guided checklist: "Your next steps: 1) Review your first synced transactions 2) Check reconciliation 3) Set up Smart Rules". Use an empty-state-as-onboarding pattern on each major page.

Minor   F-09: SPA signup page has no SEO/accessibility content

Screen: go.synder.com/signup

The signup page renders entirely client-side. Page title is just "Synder" with no descriptive content. This affects: (a) users with slow connections see a blank page, (b) screen readers get no initial content, (c) social sharing/link previews show nothing.

Rec: Implement SSR or at minimum add a loading skeleton, descriptive meta tags, and proper page title ("Create your Synder account — Free 15-day trial").

Minor   F-10: Overwhelming pricing comparison table

Screen: synder.com/pricing/

The pricing page contains 40+ feature rows in a dense comparison table, many with jargon ("Sage Intacct dimensions mapping", "COGS tracking", "Smart Rules"). New users evaluating Synder are confronted with enterprise-grade complexity before they've even tried the product.

Rec: Show a simplified "Top features" comparison (5-7 rows) by default with an "See all features" expandable. Add a "Which plan is right for me?" interactive quiz/wizard.

Minor   F-11: Cookie consent banner interrupts first experience

Screen: All pages (synder.com)

LogRocket data shows "Accept all cookies" is one of the top click events for new users. The cookie banner competes for attention with the primary CTA. On some pages it appears prominently, taking focus away from the signup action.

Rec: Use a minimal, non-intrusive cookie banner (bottom strip, not modal). Don't show it on signup/onboarding pages where user has already engaged.

Minor   F-12: Support tier differentiation is punitive for new users

Screen: synder.com/pricing/ (Support section)

Starter plan gets only "Basic (email)" support with "investigations may take over 2 weeks to begin". For new users still learning the product, this is a poor first impression — they're most likely to need help during setup, which is when support is worst.

Rec: Offer enhanced onboarding support for all plans during the first 30 days (chat access). Frame the support tiers as "ongoing support" rather than "we won't help you set up on the cheap plan."

5. Copy Audit

Problematic Copy with Suggested Rewrites

5.1 — "Choose an option" (Onboarding role selector)

❌ Current
"Choose an option"
✅ Suggested
"What best describes your business?"
or "How do you sell online?"
with options like: "I sell on marketplaces (Amazon, eBay, Etsy)", "I have an online store (Shopify, WooCommerce)", "I run a SaaS/subscription business"

Why: "Choose an option" is the most generic possible label. It tells the user nothing about what they're choosing or why it matters. This is the #1 drop-off point.

5.2 — "The final price is tailored to match your monthly transaction syncing needs within the plan limits"

❌ Current
"The final price is tailored to match your monthly transaction syncing needs within the plan limits."
✅ Suggested
"Price depends on your monthly transaction volume. Start your free trial to see your personalized price."
or add a slider: "How many transactions per month? [500] [1,000] [5,000] → Your price: $XX/mo"

Why: The current copy is corporate-speak that makes pricing feel unpredictable. "Tailored" and "within the plan limits" are vague. Users want to know what they'll pay.

5.3 — "Next step" (Onboarding CTA)

❌ Current
"Next step"
✅ Suggested
"Continue to connect your platforms →" (step 1)
"Set up your sync →" (step 2)
"Start importing →" (step 3)

Why: "Next step" provides no forward momentum or preview of what's coming. Contextual CTAs reduce anxiety and set expectations.

5.4 — "Seamless Online Sales Transactions Bookkeeping" (Hero headline)

❌ Current
"Seamless Online Sales Transactions Bookkeeping"
✅ Suggested
"Your sales, automatically in your books"
Subtitle: "Connect Shopify, Amazon, Stripe, and 30+ platforms to QuickBooks or Xero. Synder syncs every sale, fee, and refund — so your books are always right."

Why: "Online Sales Transactions Bookkeeping" is noun-stacked jargon. Lead with the outcome, not the category. Show the user what changes for them.

5.5 — Support tier copy

❌ Current
"Support requests are handled in the standard queue. Depending on complexity, investigations may take over 2 weeks to begin, starting only after express and fast track cases are resolved."
✅ Suggested
"Email support included. Need faster help? Upgrade to Scale for live chat and priority response times."

Why: The current copy actively discourages users from choosing the Starter plan by telling them their problems will wait 2+ weeks. This is honest but self-sabotaging. Frame the upgrade as an aspiration, not the base as a punishment.

Appendix: Methodology & Limitations

Data Sources

  • LogRocket Galileo AI: 5 research queries analyzing onboarding funnels, errors, page visits, and user behavior over 30-90 day windows
  • Marketing site crawl: synder.com homepage, pricing, product pages, signup flow (web_fetch)
  • App URL analysis: go.synder.com signup, auth, and onboarding endpoints

Limitations

  • No visual screenshots: Automated browser was unavailable in this environment. The internal app (go.synder.com) is a client-side SPA requiring authentication — web crawl returns minimal HTML. Visual audit would require a follow-up session with browser access.
  • LogRocket Galileo rate limits: Some queries hit iteration limits before completing full session-level analysis. The quantitative funnel data is solid; qualitative session replay insights are partial.
  • Signup flow not fully walkable: Could not create a fresh account to experience the actual signup form and immediate post-signup screens. The auth page (go.synder.com/auth) renders client-side only.
  • "Average time to first sync" query: LogRocket's second query (chatID 019d0ad8-cb22) errored out. This metric would require a custom event-based analysis.

Recommended Follow-ups

  • Browser-based visual walkthrough with screenshots of every screen
  • Session replay review of 10-15 individual user journeys (drop-offs and completions)
  • A/B test of redesigned role selection step
  • Measure time-to-first-sync with custom LogRocket events
  • Empty state audit across all post-onboarding pages
